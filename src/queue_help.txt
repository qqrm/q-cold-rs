Q-COLD queue package flow

Run a persistent dashboard daemon, stage prompts, then let the daemon open
managed task worktrees and start agents:

  qcold q-help
  qcold queue run --from queue.json --agent c1 --repo-root /path/to/repo
  qcold queue create "client queue"
  qcold queue switch <queue-tab-id>
  qcold queue list
  qcold queue append <run-id> --prompt "follow-up task"
  qcold queue stop
  qcold queue continue <run-id>
  qcold queue delete <queue-tab-id>

`qcold queue run` and mutating queue commands post to the dashboard API on
127.0.0.1:8787 by default and auto-start `qcold telegram serve --daemon` if
the daemon is not already reachable. Use `--listen <addr>` to target another
local daemon.

Preferred JSON package:

  {
    "execution_mode": "sequence",
    "layers": [
      { "name": "policy", "prompt": "shared rules for every task" }
    ],
    "default_layers": ["policy"],
    "items": [
      { "slug": "task-one", "prompt": "first task" },
      { "slug": "task-two", "prompt": "second task" }
    ]
  }

Directory or ZIP package convention:

  layers/*.md      shared prompt layers prepended to every task
  prompts/*.md     queued task prompts
  tasks/*.md       queued task prompts
  queue.json       preferred manifest when present

Graph execution is available with `execution_mode: "graph"` and per-item
`depends_on` arrays in JSON. Dependencies may reference item ids or slugs and
are stored as item ids. Queue items are still executed through managed
task open plus terminal closeout; the queue owns the agent lifecycle.

If a managed queue task reaches `closed:failed`, lands in `failed-closeout`, or
its executor exits before recording a closeout, Q-COLD starts a fresh
auto-recovery executor for that row. The recovery packet points at the failed
task record, task logs when present, and the previous failure. Q-COLD caps this
at three semantic iterations total: the original attempt plus two recovery
attempts. A successful recovery closeout advances the queue; exhausting the cap
leaves the row failed.

Remote-native execution is available with top-level `selected_execution_host:
"remote-native"` or per-item `execution_host: "remote-native"`. It requires a
`selected_remote_launcher` or per-item `remote_launcher` and currently applies
to repositories whose adapter exposes the `cargo xtask remote-agent
doctor/open/down` contract. Q-COLD exports the selected launcher as
`QCOLD_REMOTE_DEV_ENV_WRAPPER`; set `QCOLD_QUEUE_REMOTE_AGENT_LAUNCHER_ENV`
when an incubating adapter requires an additional launcher environment name.
Proxy endpoints can be supplied through `QCOLD_QUEUE_REMOTE_AGENT_LOCAL_PROXY`,
`QCOLD_QUEUE_REMOTE_AGENT_REMOTE_PROXY`, or matching JSON fields. Remote-agent
startup uses prompt files, and adapters should stage those files remotely rather
than passing full prompt bodies inline. When a remote-native launch cannot see
the opened task record or tmux session, Q-COLD schedules bounded relaunches.
Remote port-forward failures trigger best-effort remote-agent cleanup before
Q-COLD rotates to the next candidate remote proxy port.
